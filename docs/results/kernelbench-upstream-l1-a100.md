## How to read this table

_Generated by `noeris kernelbench-upstream-eval --gpu A100 --timer cuda_event` on 2026-04-11._

**Pipeline / adapter status (P0 ship state)**: The Noeris column in this
first run measures **torch-reference adapters**, not the real Noeris Triton
kernels. Each adapter in
`src/research_engine/kernelbench_upstream.py::_NOERIS_ADAPTERS` currently
calls `torch.matmul`, `torch.softmax`, etc. on the upstream fp32 shapes.
The goal of the P0 credibility pass was to stand up the honest-measurement
pipeline (upstream Model sources, fp32 dtype, cuda_event+L2-flush timer,
upstream correctness tolerance) so that the next step — swapping the
torch refs for the real Noeris Triton kernels inside each adapter — can
measure the true speedup under exactly the same conditions. That swap is
**follow-up work**, not P0.

**Rows at 1.00x / 0.99x / 0.98x** are the adapter-equals-reference rows.
They validate that the plumbing works end-to-end (fp32 shapes, timer,
allclose sampling, L2 flush) and that the fp32 baseline on A100 is ~2–5x
slower than H100 (compare the External H100 column). They do **NOT** say
anything about Noeris kernel performance yet.

**Rows worse than 1.00x** (`26_GELU: 0.12x`, `36_RMSNorm: 0.24x`) are the
ones where the adapter has to reshape the upstream 4D / large-2D tensor
into a form matching Noeris's 2D kernel interface before calling the
(reference) op. The extra data movement costs ~4–8x. This is a **real,
previously-hidden finding**: on the 4D upstream RMSNorm shape, even a
hypothetical perfect Noeris RMSNorm kernel would have to beat torch by
>4x on the reshaped 2D tensor just to break even with the upstream's
4D-native path. Noeris's previous synthetic-benchmark claim of
**rmsnorm 11.66x** was measured on 2D LLaMA shapes with matching layout
and is unlikely to transfer to the 4D upstream problem.

**Row `1_Square_matrix_multiplication_ 1.29x`** is a real speedup — the
single square matmul problem runs 1.29x faster than the upstream Model
even with torch.matmul as the adapter. Likely attributable to upstream's
wrapping of `nn.Module.__call__` adding per-call Python overhead that
`torch.matmul` directly avoids.

shapes, do_bench timer, no L2 flush):**

**Follow-up needed before shipping paper claims**:
1. Wire real Noeris triton kernels into `_NOERIS_ADAPTERS` (each adapter
   currently returns a torch reference).
2. Re-run `kernelbench-upstream-eval` on both A100 and H100 Modal.
3. For RMSNorm, decide whether to benchmark against the 4D shape with
   the reshape cost *included*, or to report the kernel-only 2D number
   with a clear disclaimer.
4. The (4096, 393216) softmax shape is outside any current Noeris bucket;
   a new bucket covering n_cols >= ~400k needs to be added to
   `triton_softmax.SOFTMAX_SHAPE_BUCKETS` before the kernel will autotune
   sensibly for this problem.
